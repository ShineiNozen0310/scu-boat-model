# 03 电控与接线

## 原始 B 站工程引脚映射

根据压缩包里的 `Hardware/*.c` 和 `User/main.c`，原项目使用的是 STM32F103，核心引脚如下：

| 功能 | 引脚 | 外设 | 来源 |
| --- | --- | --- | --- |
| 红外接收 | `PA0` | `TIM2 CH1` 输入捕获 | `Hardware/IR.c` |
| 前/后/左/右状态灯 | `PA1` - `PA4` | GPIO 输出 | `Hardware/Led.c` |
| 舵机 PWM | `PA6` | `TIM3 CH1` | `Hardware/Servo.c` |
| 电机 PWM | `PA8` | `TIM1 CH1` | `Hardware/Motor.c` |
| 电机方向 | `PB12`, `PB13` | GPIO 输出 | `Hardware/Motor.c` |

## 当前仓库建议的电控分层

- `firmware/cubemx/`：GPIO、TIM、ADC、NVIC、时钟初始化
- `firmware/app/drivers/`：红外协议解析、采集队列、板级 HAL 接口
- `firmware/app/control/`：档位、油门、舵角控制
- `firmware/app/safety/`：失联保护、低压、急停策略

## 低压检测建议

现在 `MainAppPlatform` 里已经预留了 `battery_is_low` 回调，建议实际工程里补一个简单方案：

- 电池分压到 ADC
- 软件里做多次采样平均
- 低于阈值后先报警，再禁止推进

## 结合宣讲 PPT 可直接吸收的电控点

### 1. PWM 统一基准

宣讲 PPT 里提到舵机常见控制周期是 `20ms`，这一点对当前工程是有帮助的，但项目里建议继续统一成更明确的工程口径：

- 舵机 / 双向电调统一按 `50Hz` PWM 设计
- 统一使用 `1000us / 1500us / 2000us` 作为最小 / 中位 / 最大控制脉宽
- 船端所有执行器都通过板级 HAL 输出这一套标准脉宽，不在业务层散落换算

这样做的好处是:

- 舵机、双向 ESC 和云台轴都能共用同一套标定方法
- 后面换 MCU、换定时器、换发射端输入方式时，不需要改控制逻辑

### 2. DIY 发射端的 ADC 输入

宣讲 PPT 里“摇杆模拟量 -> ADC -> 控制量”的思路是成立的，这对你后面手搓遥控器有直接帮助：

- 双轴摇杆优先走 ADC 采样
- 软件里先做中位校准
- 再做死区、限幅和简单滤波
- 最后映射成 `throttle / rudder / turret_yaw / turret_pitch`

建议工程上固定成下面这条链路:

`ADC 原始值 -> 去抖/均值 -> 中位校准 -> 死区 -> 百分比控制量 -> 无线发包`

### 3. 无线模块接口选择

宣讲 PPT 把 `LoRa / 蓝牙 / Wi-Fi / NRF24L01` 都列成“通信模块”，这个分类本身没问题，但对当前工程最重要的结论是：

- 如果目标是尽快做出预算可用版, 优先选 `UART` 透明模块
- 如果目标是最终稳态主链路, 优先选成熟 RC 协议链路
- 不要为了“模块便宜”就选一个会把软件集成成本拉高很多的 `SPI` 射频模块

对你当前仓库, 这意味着:

- 预算链路更适合 `E220 UART`
- 成熟链路更适合 `ELRS -> CRSF`

## 上电检查顺序

1. 检查电池电压和极性。
2. 断开螺旋桨或让船体离水，先做空载测试。
3. 确认舵机初始化后回中。
4. 观察 LED 和 OLED 是否与控制状态一致。
5. 最后再验证电机正反转是否和逻辑定义一致。
