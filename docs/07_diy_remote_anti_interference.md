# DIY 遥控器抗干扰方案

更新日期: 2026-03-13

## 1. 结论先行

如果现在遥控器必须手搓, 想把抗干扰能力放在第一优先级, 推荐采用下面这条路线:

- 手搓的是: 外壳、摇杆、拨杆、按键、电源、UI、主控板
- 不手搓的是: 空口协议和射频底层
- 主链路建议: `DIY 发射端 -> ELRS 发射模块/成熟 RF 模块 -> 船端接收机 -> CRSF -> STM32`

这和仓库当前固件方向一致:

- 船端已经支持 `CRSF` 输入, 入口在 `firmware/app/main_app.c`
- `boat_crsf.c` 已经实现了解包、CRC 校验和通道映射
- `boat_radio_protocol.*` 更适合做实验或短链路验证, 不能把它误认为“高抗干扰无线链路”

一句话: 可以自己做遥控器, 但不要自己发明主无线协议。

## 2. 为什么不建议自定义裸 RF

抗干扰不是“包里加一个 CRC”就够了。真正决定链路稳定性的, 主要是这些能力:

- 跳频或成熟的频率管理
- 发射功率、空口速率、接收灵敏度之间的平衡
- 接收机恢复、重同步和丢包处理
- 链路质量指示与失联保护
- 天线、供电、地线和电机噪声隔离

当前 `boat_radio_protocol.*` 只有:

- 固定长度帧
- 同步字
- CRC-8
- 序号字段

这足够做“应用层命令封装”, 但不足以单独承担“复杂环境下高抗干扰遥控链路”的要求。

## 3. 推荐架构

### 3.1 发射端

建议结构:

1. 主控 MCU 读取摇杆、拨杆、旋钮和按键
2. 本地完成校准、滤波、死区和限幅
3. 主控输出标准控制数据给成熟 RF 链路
4. 由成熟 RF 链路发往船端

推荐分层:

- 人机输入层: `STM32` 或 `ESP32`
- 射频层: 优先复用成熟 `ELRS` 发射模块或成熟开源方案
- UI 层: OLED / 电量显示 / 蜂鸣器 / 链路状态指示

建议不要让主控 MCU 直接裸控一个低级 RF 模块去“现设计现发包”, 除非你愿意单独做很长一轮无线调试。

### 3.2 船端

建议结构:

- 接收机输出 `CRSF UART`
- STM32 只负责解析 `CRSF` 并映射为 `BoatCommand`
- ESC、舵机、水炮电机等执行器与接收链路做供电和布线隔离

这个仓库已经具备这层结构, 继续沿用即可。

## 4. 发射端必须做好的硬件点

### 4.1 电源

- 射频供电和摇杆 ADC 参考电源不要直接挂在脏电源上
- 电池输出后先做降压, 再给 RF/模拟输入分别做滤波
- 至少预留大电容 + 0.1uF 去耦
- 如果主控 ADC 抖动明显, 单独给摇杆参考侧做 RC 滤波或 LDO

### 4.2 输入器件

- 摇杆优先选霍尔摇杆, 其次再考虑普通电位器
- 关键开关用带明确段位感的拨杆, 不要用手感差的小按键代替
- 中位量大的一路要做软件校准和死区

### 4.3 PCB / 布线

- RF 模块远离 DC-DC、电源电感、屏幕排线和蜂鸣器
- 模拟摇杆线短、远离高 di/dt 回路
- 地线优先完整回流, 不要让 RF 和大电流负载共用细长地线
- 天线附近不要压金属件、整块电池和大面积铜皮

### 4.4 天线

- 天线要露出塑料外壳或贴近非金属区域
- 手握姿势不能长期把天线完全包住
- 使用模块原配天线或阻抗匹配明确的天线, 不要随便焊一截线凑合

## 5. 船端必须做好的抗干扰点

很多“遥控失控”不是空口不行, 而是船上的电机噪声把接收机和 MCU 一起带崩了。

### 5.1 供电隔离

- 接收机和 MCU 由稳定 `5V UBEC` 供电
- 舵机和水炮如果电流冲击大, 不要和接收机走一条细供电线
- 电机侧和逻辑侧地线要短而粗, 但走线分区明确

### 5.2 电机噪声

- 有刷电机两端加抑制电容
- 电机线尽量短, 与接收天线分开
- ESC 到电机的大电流线不要和接收机天线平行贴走
- 必要时在电机线或电源线加磁环

### 5.3 接收机安装

- 接收机天线远离电机、ESC、电池和金属喷水结构
- 不要把天线塞在完全封闭的金属/导电屏蔽空间里
- 防水可以做, 但不要为了防水把 RF 完全闷死

## 6. 固件层面必须保留的安全策略

无论用什么发射端, 船端都必须保留这些策略:

- 帧超时后自动回中 / 零油门
- 急停通道独立处理
- 链路恢复后不要瞬间恢复到上次大油门
- 输入做限幅、死区和斜坡

当前仓库已经有这些基础:

- `BOAT_COMMAND_TIMEOUT_MS` 定义了失联超时
- `MainApp_ApplyCommand()` 会刷新接收时间
- `BoatSafety_Tick()` 负责失联保护
- `MainApp_SetEmergencyStop()` 支持急停

如果后面自定义发射端, 船端优先走 `CRSF` 入口, 不要退回红外和简化裸包链路作为主方案。

## 7. 如果你被迫完全自制无线链路

只有在下面条件都满足时, 才建议自己做空口:

- 接受明显更长的调试周期
- 接受大量台架测试和水面测试
- 接受“第一版不如成熟链路稳”这个结果

最低要求至少要有:

- 明确的包头、长度、版本号
- `CRC-16` 或更强校验
- 单调递增序号
- 丢包检测和重复包丢弃
- 发射周期固定
- 多帧冗余发送
- RSSI / LQ 观测手段
- 严格的失联超时和回中策略

即使做到这些, 也仍然不一定比成熟链路更稳。

## 8. 推荐的工程决策

对这个项目, 推荐直接定下下面三条:

1. 手搓遥控器只做“人机界面”和结构件, 不自研主无线协议。
2. 船端主输入统一收敛到 `CRSF`。
3. 抗干扰验证优先做供电、布线、天线和失联保护, 再谈更复杂的功能。

## 9. 测试清单

### 9.1 台架

- 摇杆中位漂移测试
- 发射端低电压测试
- 电机空转 / 加负载时的接收稳定性
- 急停是否独立可靠
- 断开发射端后多快进入零油门

### 9.2 环境

- 室内近距离抗噪声测试
- 室外开阔地距离测试
- 船盖关闭后的链路对比
- 电池满电 / 半电 / 低电时的链路对比

### 9.3 下水

- 低速巡航链路稳定性
- 大油门转向时接收稳定性
- 水炮工作时接收稳定性
- 失联后是否稳定切安全态

## 10. 如果必须把整船压到 800 元内

如果你现在最硬的约束是预算, 那就不能再把“成熟 ELRS 主链路”当默认前提了。

这时更现实的做法是:

- 发射端主控: `ESP32`
- 发射端输入: 双轴摇杆 + 拨杆/按键
- 无线模块: 一对 `E220 UART LoRa` 透明传输模块
- 船端固件入口: `MainApp_OnRadioPacket(...)`
- 应用层数据帧: `boat_radio_protocol.*`

这条路线的优点:

- 无线链路本身仍能压在低价位
- 整船 BOM 更容易压到 800 元内
- 集成难度比自己裸写 SPI 射频驱动低
- 仓库里已经有 `OnRadioPacket` 和固定长度控制帧入口

这条路线的代价:

- 抗干扰能力和链路成熟度不如 `ELRS -> CRSF`
- 发射周期、丢包处理和测试要你自己盯
- 后面想做标准化遥测和链路配置会比 `CRSF` 麻烦

所以它适合:

- 先把自制遥控器做出来
- 先把预算打下来
- 接受“这是预算原型链路, 不是最终稳态方案”

## 11. 当前建议

如果你现在要尽快推进, 最稳的落地方式是:

- 遥控器外壳、摇杆、电源、UI 自己做
- 无线主链路复用成熟 `ELRS -> CRSF`
- 船端继续沿用本仓库现有 `CRSF` 接口

这样“遥控器是自己做的”这件事可以成立, 同时抗干扰能力不会因为自定义裸链路被明显拉低。
